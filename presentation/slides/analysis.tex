\begin{frame}
\frametitle{Vulnerability: Priority-Based Scheduling}

FreeRTOS uses a priority-based preemptive scheduler, which is inherently unfair and vulnerable to timing attacks.

\vspace{0.3cm}
\begin{center}
\resizebox{0.70\textwidth}{!}{\input{figures/priority_sched.tex}}
\end{center}

\vspace{0.3cm}
\textbf{The channel:} A high-priority task signals \texttt{0} or \texttt{1} by
sleeping or running, and the low-priority task measures its own wake time.

\vspace{0.2cm}
\tiny\fullcite{chen2019}

\end{frame}

\begin{frame}
\frametitle{Possible Solutions}
We considered several approaches to address priority-based timing vulnerabilities.

\begin{itemize}
  \item \textbf{Fixed execution time:} Each priority would have a known fixed execution time and scheduling. \textcolor{red}{Significant idle time}
  \item \textbf{Introduction of Domains:}
    \begin{itemize}
      \item A domain is a group of tasks, which the scheduler treats as a
        single environment
      \item Only partitioning across domain boundaries allows for intra-domain
        channels
      \item Requires the notion of time slices, which is a fixed execution time
        serving as the smallest unit of "time" for the scheduler
      \item Should domains be dynamic or static?
    \end{itemize}
\end{itemize}

\end{frame}

\begin{frame}
\frametitle{Design: Domain Scheduling}
\textbf{Chosen approach:} Domain scheduling
\vspace{0.1cm}
\begin{center}
\centering
\resizebox{0.5\textwidth}{!}{\input{figures/dynamic_domain_sched.tex}}
\end{center}

\vspace{0.2cm}
\begin{itemize}
  \item Initially, all time slices are given to Domain 0
  \item Domain 0 gives up fixed time slices to other domains on creation
  \item Tasks within a domain use standard priority scheduling
  \item Allows for a trade-off between the throughput (priority) and
    partitioning (domain)
\end{itemize}

\end{frame}
