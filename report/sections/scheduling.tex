\section{Scheduling}\label{sec:Scheduling} % (fold)

\subsection{The FreeRTOS scheduler} % (fold)
FreeRTOS provides different scheduling algorithms based on configurations and
number of cores available on the system. By default on a single core FreeRTOS
makes use of a fixed-priority preemptive scheduling policy, with round-robin
time-slicing of equal priority tasks. \cite{FreeRTOSschedulingsinglecore2026}

As such, FreeRTOS always attempts to schedule the highest priority task and
possibly starves lower priority tasks. Items of the same priority run until a
tick interrupt at which point the time slice is up. This type of scheduling
inherently includes timing side-channels as the higher priority tasks inhibit
lower priority tasks from running. \cite{kadloor2016, chen2019} This could be
used as a covert channel, where the higher priority task could signal either a
0 or 1 by simply going to sleep and the lower priority task measures its own
execution time.

Therefore, for FreeRTOS to provide timing protection the scheduler must see
modifications.

\subsection{Closing Scheduling Based Timing Channels}

There are a few approaches to closing the different side channels each with
their own pros and cons. A general attribute among all of them is, that
starvation must not be possible, and as such the current approach to priority
based scheduling is infeasible. 

\paragraph{Removing time:} One approach to solve a lot of the timing channels
would be to disallow tasks to read the current time. This in theory would
remove the scheduling based timing channel, but this solution would most likely
attributed quite a significant decrease in the possibilities of tasks, as any
action, which could leak time information to the tasks would need to be
prohibited. As such, this approach was not considered further.

\paragraph{Protection within priority:} If the goal is to keep a sense of
priority based scheduling, then it should be possible to guarantee timing
protection for tasks of same priority or tasks of different priorities. Say for
example, we for each priority have a known array which constitutes the time
slices for the given priority. When the scheduler attempts to schedule a task,
then it prioritizes the higher priority array if a task is ready within, and
uses time slice based scheduling within the priority. An example of this can be
seen visualized in Figure~\ref{fig:prio_sched}, where tasks A and B are within
the same priority. Here, if we assume that A and B are runnable, then they
would be scheduled before the lower priority C and D. This would allow C and D
to gather information about the runnability of the entire High priority noting
that either A or B was runnable if C or D are delayed. However, within the same
domain tasks A and B can not gather information about each other. Say for
example A gets to run, then it does not know whether task B is runnable or not,
as we are always running the entire time slice given defined by the
aforementioned array.

This approach would by no means close all timing channels, as mentioned cross
priorities could still leak timing information. Other constraints with this
approach is, that we are limiting the number of possible tasks for given
priorities. In the previous example 4 time slices were provided within the High
and Low priorities, and we would not be able to fill more than these 4 time
slices as we would then change the total time slice when running a specific
priority and thus a task within the given priority could gather information
about the total number of time slices. Another issue would be, that we can't
simply give up CPU time. When a given tasks wishes to sleep for a short
duration, we must replace the time slice with idle time, which significantly
reduces overall CPU utilization. Once a task is completely finished it is
possible to then take back the time slice, but the new task that would be
scheduled in this slot would then gather the information that the previous task
finished its execution.

\paragraph{Domain scheduling:}
Another approach is the one seen in \cite{ge2019}, which is the domain level
scheduler. The general idea is quite similar to the approach seen before, but
the idea of priorities is removed entirely and instead we focus on
domains. Each domain can have multiple tasks within, where we do not care about
timing channels and only between different domains do we care about timing
protection. In Figure~\ref{fig:static_domain_sched} an illustration of this
scheduler can be seen. The idea is, that at compile time, we know the different
domains and attribute to each some amount of time slices (in the example that
would be 4 for both domain 1 and 2). The domain scheduler then takes care of
scheduling the domain, and guarantees timing protection only between the
different domains. This approach is still inhibited by the need to know the
total number of time slices at compile time, and further it also needs the
number of time slices for each domain at compile time. However, where this
benefits over the previous approach is, that within each domain there is a lot
more freedom to do as pleased. For example, one could imagine having a regular
round robin scheduler within a domain, which then allows more CPU utilization
within this specific domain and less unused resources.  

An extension of this approach would be the one seen in
Figure~\ref{fig:dynamic_domain_sched}, which closely follows the capability
based scheduler in \cite{karlsson2025}. The idea is, that within a specific
domain, we can give up some of the attributed time to create a new subdomain.
This would mean that there is timing protection between subdomain 1 and the
rest of domain 1\footnote{Other than creation and finishing assuming a domain
  is allowed to give back the time slices upon finishing, which would leak
timing information}. In theory it should be possible to give up these time
slices back to the domain level scheduler, which would then treat this as a new
domain. Whether or not some timing channels persists after creation remains to
be seen.

This approach seems by far the most flexible and most robust in terms of real
world applications. As such, this thesis attempts to implement this domain
scheduler within the FreeRTOS kernel.

It is important to note, that this approach does not remove all timing
channels. Each domain uses a round robin scheduler, which as mentioned
previously is susceptible to timing channels. The idea is instead, that we
allow such channels in tasks where it is not relevant for protection and
throughput is valued higher. Instead, when we explicitly want a safe execution
environment we can create a domain for this, which is then protected after
initialization.

\subsection{Implementation}

\subsubsection{Extending the api}\label{sec:api}

To implement the domain level scheduler we have first incorporate the idea into
the existing FreeRTOS framework. With FreeRTOS MPU, we are able to create
unprivileged tasks which have restricted access to syscalls and provide them
with a memory space that is partitioned. For this, the general FreeRTOS api is
extended with a multitude of new functions most importantly for this project is
the \texttt{xTaskCreateRestricted}.\cite{MemoryProtectionUnit2026} The idea
behind the domain scheduling fits with the purpose of memory protection, and as
such it seems the obvious choice for modification to extend it with the domain
functionality. The signature of the aforementioned function is provided in the
listing below:

\begin{lstlisting}[language={c}, caption={xTaskCreateRestricted function signature}, label={lst:create_restricted}]
typedef struct xMEMORY_REGION
{
    void * pvBaseAddress;
    uint32_t ulLengthInBytes;
    uint32_t ulParameters;
} MemoryRegion_t;

typedef struct xTASK_PARAMETERS
{
    TaskFunction_t pvTaskCode;
    const char * pcName;
    configSTACK_DEPTH_TYPE usStackDepth;
    void * pvParameters;
    UBaseType_t uxPriority;
    StackType_t * puxStackBuffer;
    MemoryRegion_t xRegions[ portNUM_CONFIGURABLE_REGIONS ];
    #if ( 
      ( portUSING_MPU_WRAPPERS == 1 ) && 
      ( configSUPPORT_STATIC_ALLOCATION == 1 ) 
    )
        StaticTask_t * const pxTaskBuffer;
    #endif
} TaskParameters_t;

BaseType_t xTaskCreateRestricted( 
  const TaskParameters_t * const pxTaskDefinition, 
  TaskHandle_t * pxCreatedTask 
) PRIVILEGED_FUNCTION;
\end{lstlisting}

That is, we can create a task which has access to a number of memory regions
defined by the region start address (pvBaseAddress), the region size in bytes
(ulLengthInBytes) and the access permissions for the region (ulParameters). The
function parameter uxPriority is also of importance. By default this function
creates tasks running in user mode, which restricts access to specific system
calls and any task created running in privileged mode is able to access the
entire RAM. By default, tasks running in user mode are able to create other
tasks running in user mode, and this is the desired behaviour for domain
scheduling.

The idea is to extend this function signature as follows:

\begin{lstlisting}[language={c}, caption={Extended parameters and signature}, label={lst:extended_create_restricted}]
...

typedef struct xDOMAIN_PARAMETERS
{
  uint32_t ulSliceIndex;
  uint32_t ulSliceLength;
} DomainParameters_t;

typedef struct xTASK_PARAMETERS
{
  ...
    #if ( configENABLED_DOMAINS == 1 )
        DomainParameters_t * xDomainParameters; 
    #endif
} TaskParameters_t;

BaseType_t xTaskCreateRestricted( 
  const TaskParameters_t * const pxTaskDefinition, 
  TaskHandle_t * pxCreatedTask 
) PRIVILEGED_FUNCTION;
\end{lstlisting}

A task with the capability to create a new restricted task would then have to
provide the index in the global time slice array, along with the length of the
new domain that will be created. If no xDomainParameters is passed to the
function (i.e. the pointer is NULL) this will be treated as creating a task
within the same domain, and will be handled by the per domain priority based
scheduler.

With this, no more of the external api has to be changed.

\subsubsection{New configuration flags}
To enable the functionality of domain level scheduling, a few new compile flags
need to be introduced.

\paragraph{configENABLE\_DOMAINS} A binary flag to indicate whether or not to
use the domain level scheduler alongside the default priority based scheduler.

\paragraph{configNUM\_TIME\_SLICES} The total number of time slices for the
domain scheduler to take into account. As mentioned before this flag also
constitutes the maximum number of possible domains, such that any domain
scheduling decisions can be done in constant time.

\paragraph{configNUM\_TICKS\_PER\_SLICE} The number of timer interrupt ticks to
constitute a single time slice. Say that the ticks is defined as
\texttt{\#define configTICK\_RATE\_HZ ((TickType\_t) 1000)} with \texttt{\#define
configNUM\_TICKS\_PER\_SLICE (5)}, this would mean that a single time slice is
equal to $\text{TIME\_SLICE} = \frac{1s}{1000} \cdot 5 = 5ms$.

\subsubsection{Domain switching}

To keep with the fundamentals of FreeRTOS the idea is to keep the functionality
of the priority based scheduler within FreeRTOS but internally in a domain.
That is, within a specific domain the same priority based scheduler is
available. This in turn means that if no new domains are created, then the
functionality of the scheduler should be nearly unchanged\footnote{There will
be some overhead with the implementation of course.}.

FreeRTOS makes use of a single static file \texttt{tasks.c} for the development
of the priority based scheduler. In this thesis I am only focused on the single
core implementation and while work on the multicore version would be
interesting, that is left for future work.

In the MPU version of FreeRTOS when a machine timer interrupt occurs it is
handled in the following manner:

\begin{lstlisting}[language={[RISC-V]Assembler}, caption={FreeRTOS mtimer interrupt handler}, label={lst:interrupt}]
freertos_risc_v_mtimer_interrupt_handler:
    portcontextSAVE_INTERRUPT_CONTEXT
    portUPDATE_MTIMER_COMPARE_REGISTER
    call xTaskIncrementTick
    beqz a0, exit_without_context_switch 
    call vTaskSwitchContext
\end{lstlisting}

It increments a global tick, and switches the task context. However, FreeRTOS
only increments the global \texttt{xTickCount} if the priority based scheduler
is running and not suspended for critical handling. This means, we have to
introduce our own proxy for Ticks, as the per domain \texttt{xTickCount} would
leak information about other scheduler states and possibly delay the domain
switching. For our purposes we can hook into the pre-existing
\texttt{xTaskIncrementTick} function, and check for a domain switch.

\begin{lstlisting}[language={c}, caption={Custom hook for xTaskIncrementTick}, label={lst:inc_tick}]
/* Static variable to contain the current Domain Tick */

PRIVILEGED_DATA static size_t pxCurrentDomainIndex = 0;
PRIVILEGED_DATA static volatile TickType_t xEffectiveTick = ( TickType_t ) configINITIAL_TICK_COUNT;
PRIVILEGED_DATA static volatile TickType_t xDomainTick = ( TickType_t ) 0;

BaseType_t xTaskIncrementTick( void ) {
...
const TickType_t xConstTickCount = xEffectiveTick + ( TickType_t ) 1;

/* Increment the effective tick. This is different than xTickCount,
 * which does not increment if the scheduler is in a critical section.
 * The Domain scheduler does not care about critical section */
xEffectiveTick  = xConstTickCount;

/* Update the Domain tick in regards to the effective tick */
xDomainTick = (xEffectiveTick / configNUM_TICKS_PER_SLICE) % configNUM_TIME_SLICES;

/* Time slot expired or wrap-around */
if( xDomainTick >= pxCurrentDomainIndex + xDomains[pxCurrentDomainIndex].uxLength
    || xDomainTick < pxCurrentDomainIndex )
{
    /* Save current task to logical domain info */
    xDomainInfo[xDomains[pxCurrentDomainIndex].uxDomainID].pxPreviousTCB = pxCurrentTCB;

    xSwitchRequired = pdTRUE;

    /* Flush on domain switch */
    temporal_fence_t();

    /* Contiguous partition: next block starts exactly here */
    pxCurrentDomainIndex += xDomains[pxCurrentDomainIndex].uxLength;
    if( pxCurrentDomainIndex >= configNUM_TIME_SLICES )
    {
        pxCurrentDomainIndex = 0;
    }

    /* Restore previous task from logical domain info */
    TCB_t * previous_tcb = \ 
      xDomainInfo[xDomains[pxCurrentDomainIndex].uxDomainID].pxPreviousTCB;
    /* First time, use idle task */
    if (previous_tcb == NULL) {
        pxCurrentTCB = xIdleTaskHandles[ 0 ];
    } else {
        pxCurrentTCB = previous_tcb;
    }
}
...
}
\end{lstlisting}

As such, whenever there is a machine timer tick, we calculate in constant time
the time slice (xDomainTick) and check if we should do a domain switch. If we
find that we should schedule a new domain, then we switch the data to the new
domain and perform a temporal flush to clear the microarchitectural state. The
flush happens in such a manner, that any data accessed in regards to the new
domain is non-interfering with the previous domain. Once we have decided that a
switch should occur, only data that is within the incoming domain is retrieved
following the flush, this means that the new domains execution will not be
based on the microarchitectural state of the previous domain.

Note this comes with some constraints. Most notably we are defining a time
slice in relation to the machine timer interrupt timing, which is configurable
in FreeRTOS through the \texttt{configTICK\_RATE\_HZ}. To make this more
flexible, the \texttt{configNUM\_TICKS\_PER\_SLICE} as to not limit the domain
level scheduler with respect to the inter domain priority based scheduler.

\subsubsection{Constant time domain creation}
To keep track of the different domains new global constants are defined:
\begin{lstlisting}[language={c}, caption={Domain timeline}, label={lst:domain_def}]
typedef struct DomainBlock {
  UBaseType_t uxDomainID; /* This time slices domain ID */
  size_t uxLength; /* Length of this domain */
  size_t uxNext;
} DomainBlock_t;

/* Timeline over configNUM_TIME_SLICES, indexed by the start of a block */
PRIVILEGED_DATA static DomainBlock_t xDomains[configNUM_TIME_SLICES] = {
    [0] = {.uxDomainID = 0, .uxLength = configNUM_TIME_SLICES, .uxNext = 0}
};
\end{lstlisting}

For now the main data structure introduced is an array, namely xDomains,
storing the timeline. This array is indexable by the start time of the current
group of time slices and contains information about the current group of domain
slices, such as the length. Two example states of this array can be seen
visualized in Figure~\ref{fig:example_timeline}. For the case where Domain 0
owns all time slices xDomains would only contain a single valid index, namely
the 0 index, which stores that it is domain 0, that owns the group of time
slices and the group has a length of 6.

This way, our previous domain switching can be directly indexed once we find
that a new domain switch needs to happen providing constant time retrieval of
the next domain.

At first glance domain creation also seems simple. In
Figure~\ref{fig:example_a} there are three main cases for domain creation.
Either we create domains by giving up time slices at the start or the end of
our own time slice, which would simply create two separate domains with no
"split" domain. If instead a domain is created in the middle of the current
domain, then the current domain is split and might be scheduled several times
during a single round trip of the time slices. Such an example can be seen in
Figure~\ref{fig:example_b}, where Domain 0 runs at separate times of execution.
This poses an issue when wanting to create yet another domain. As mentioned in
Section~\ref{sec:api}, a user of the OS would provide the time slice for which
to start a new domain and the length of the domain. Given that not all indices
throughout the domain are defined, we would need to scan the array from left to
right to figure out, if the current domain truly does "own" the given time
slices the user is attempting to access. If a user wants to create a new domain
from Domain 0s time slice 4-6 (0 indexed), then the naive method would loop
from the start of the array until reaching the group of time slices, which
contains these time slices, namely indices 4-8. This execution is not constant
time with respect to what domain 0 should know as the execution time of the
loop depends on how many other domains are created before no matter which
domain owns those time slices.

To solve this issue the \texttt{uxNext} field was introduced, which contains
the index of the next group of time slices owned by the current domain. This
field can then be used for the looping, when finding the encompassing group of
time slices giving us execution that is only dependent on information available
to the current domain. In the case that there is no next domain the
\texttt{SIZE\_MAX} is used.

With this in place, it allows finding the encompassing domain as follows:
\begin{lstlisting}[language={c}]
static BaseType_t prvFindContainingBlock(
    size_t uxFirstBlock,
    size_t start,
    size_t length,
    size_t * puxPrevBlock,
    size_t * puxBlock)
{
    size_t prev = SIZE_MAX;

    for (size_t current = uxFirstBlock; current != SIZE_MAX; current = xDomains[current].uxNext)
    {
        size_t blockEnd = current + xDomains[current].uxLength;

        if (start >= current && (start + length) <= blockEnd)
        {
            *puxPrevBlock = prev;
            *puxBlock = current;
            return pdTRUE;
        }

        prev = current;
    }

    return pdFAIL;
}
\end{lstlisting}

Once we have found the encompassing group of time slices we have to create the
new domain, which is done via index arithmetic as seen below: 

\begin{lstlisting}[language={c}]
...
/* Right fragment */
size_t rightStart = SIZE_MAX;
if (start + length < blockEnd)
{
    rightStart = start + length;
    xDomains[rightStart].uxDomainID = uxCurrentDomainID;
    xDomains[rightStart].uxLength = blockEnd - rightStart;
    xDomains[rightStart].uxNext = xDomains[blockStart].uxNext;
}

/* Left fragment: shrink existing block */
if (start > blockStart)
{
    xDomains[blockStart].uxLength = start - blockStart;
    if (rightStart != SIZE_MAX)
    {
        xDomains[blockStart].uxNext = rightStart;
    }
}
else /* No left fragment: remove blockStart from parent's list */
{
    size_t spliceNext = (rightStart != SIZE_MAX) ? rightStart : xDomains[blockStart].uxNext;
    if (prev == SIZE_MAX)
    {
        xDomainInfo[uxCurrentDomainID].uxFirstBlock = spliceNext;
    }
    else
    {
        xDomains[prev].uxNext = spliceNext;
    }
}
...
\end{lstlisting}

We first check, if we would create a new right fragment of the current domain.
It would exists if we are attempting to create a new domain, which start ends
before the current domain. If such a fragment exists, then we update the index
in \texttt{xDomains} to track it.

Next, we can check if a left fragment is created, and simply shrink the
encompassing time slice group and updating the \texttt{uxNext} field indicating
that we have been split and the next block owned by the current domain is that
of the new right fragment.

If there is no left created, then we are creating a new domain at the start of
the encompassing block, so we remove the encompassing block start from the
linked list, and update it to point to the right fragment if it was created or
the next block. In the case that there is no next block, then it was previously
set to SIZE\_MAX indicating again, that there is no next block.

At last, we have to update the \texttt{xDomains} to include the newly created
domain, which is simply done as follows:

\begin{lstlisting}[language={c}]
...
UBaseType_t uxChildDomainID = uxNextDomainID++;
xDomains[start].uxDomainID = uxChildDomainID;
xDomains[start].uxLength = length;
xDomains[start].uxNext = SIZE_MAX;

xDomainInfo[uxChildDomainID].pxPreviousTCB = NULL;
xDomainInfo[uxChildDomainID].uxFirstBlock = start;
...
\end{lstlisting}

The global xDomainInfo is the array used for scheduling the TCB within each
domains, and is indexable by the domain id rather than the start of the time
slice block.

\subsubsection{Per domain scheduling}
For the domain level scheduler to work with inter domain priority based
schedulers, we need to add another dimension to all of the items. That is, all
variables that previously were a constant would have to be an array with length
equal to the number of time slices and same extension for previous arrays. All
accesses to these constants/arrays would have to follow a similar
transformation, where the currently scheduled domain would choose the correct
values for the priority based scheduler to function correctly. A few examples
of these transformations are shown below:
\begin{lstlisting}[language={c}, caption={Transforming the priority based scheduler}, label={lst:prio_sched}]
/* Non-domain scheduler implementation*/
PRIVILEGED_DATA static volatile BaseType_t xSchedulerRunning = pdFALSE;
PRIVILEGED_DATA static volatile TickType_t xPendedTicks = ( TickType_t ) 0U;

/* Domain scheduler equivalent */
PRIVILEGED_DATA static volatile BaseType_t xSchedulerRunning[configNUM_TIME_SLICES] = { pdFALSE };
PRIVILEGED_DATA static volatile TickType_t xPendedTicks[configNUM_TIME_SLICES] = {( TickType_t ) 0U};
\end{lstlisting}

Note, that while the first element is initialized here, the initialization of
the full items are moved to a separate function, as all of the schedulers would
have to be initialized in the same fashion as the default scheduler.

With this, the general foundation of the domain level scheduler is functional,
and the standard \texttt{mpu\_blinky.c} example works given all tasks run
within the same domain.

\subsubsection{Cross domain communication}
As it stands, the FreeRTOS api has next to no information about the different
domains, meaning that items such as the Queue used for message transferring
between domains is not available. Generally this is a good thing, as by default
there is domain isolation, as any internal handling within tasks.c makes use of
the current domain. This restriction is quite strict, and one might assume that
cross-domain communication should be possible. In FreeRTOS communication
happens through Queues. The general workflow for Queue communication in the MPU
version of FreeRTOS is as follows:

\begin{itemize}
  \item Create Queue kernel object in a priority task.
  \item Grant access to the Queue object to any user level tasks, that should communicate through the Queue.
  \item One task sends messages through \texttt{xQueueSend}.
  \item Another task receives messages through \texttt{xQueueReceive}.
\end{itemize}

For the cross domain communication the workflow should be the same, but it is
the internal handling that has to change. Currently the internal handling of
the \texttt{xQueueSend} is:
\begin{itemize}
  \item Check there is room in the queue for the message.
  \item Check if there is a task waiting to receive on the Queue.
    \begin{itemize}
      \item If yes, move the task to the ready list.
      \item If no, do nothing.
    \end{itemize}
\end{itemize}

This is generally also fine, except that when moving a task to the ready list,
the task might be outside the domain that is currently running. When the Queue
moves a task to the ready list, it does so through the function
\texttt{xTaskRemoveFromEventList}. This function takes a linked list of tasks,
namely the tasks waiting on the queue, removes the first task from the list
and moves the given task into the ready list if the scheduler is suspended and
otherwise moves it to the \texttt{xPendingReadyList} to wait for the next
scheduler suspension before moving into the true ready list. If the new task
was found to have a higher priority, then notify that there should be a context
switch.

This is where we have to make changes to allow for cross-domain communication.
First, each task needs some way of tracking the domain it resides within, which
is easily done by extending the \texttt{TCB\_t} struct with an extra field:
\begin{lstlisting}[language={c}]
typedef struct tskTaskControlBlock
{
    ...
    #if ( configENABLE_DOMAINS == 1 )
        UBaseType_t uxDomainID;
    #endif
} tskTCB;

typedef tskTCB TCB_t;
\end{lstlisting}

The Queue can now reference this DomainID when modifying the
priority-schedulers and as such adds the tasks to the correct arrays.

Given this communication, we can now create the \texttt{mpu\_blinky.c} example
with different domains communicating. It is important to note here, that there
of course is introduced some timing leakage between these two domains, as they
can time the response time of different messages. Here it is up to the
individual domains to respond in a deterministic manner as to not leak any
secret information.
